# Registry index schema

`registry/index.json` is the machine-readable catalog the kasas dashboard consumes
to list and install community plugins. It is generated by `kasas-plugins index` from
the `plugins/` directory and contains only plugins that pass the
[submission gate](submission-guidelines.md).

## Where to fetch it

- **Canonical (source of truth):** the committed file on `main`
  `https://raw.githubusercontent.com/paulmeier/kasas-plugins/main/registry/index.json`
- **CDN mirror (optional):** GitHub Pages
  `https://paulmeier.github.io/kasas-plugins/index.json`

CI enforces (via `index --check`) that the committed file always matches the
plugins in the repo, so the raw endpoint is authoritative.

## Document shape

```jsonc
{
  "schema_version": 1,                       // bump on a breaking format change
  "generated_at": "2026-06-09T13:23:48Z",    // UTC; informational only
  "repository": "https://github.com/paulmeier/kasas-plugins",
  "plugins": [
    {
      "name": "coffee-budget",
      "version": "0.1.0",
      "description": "Labels transactions whose description matches a keyword.",
      "author": "kasas community",
      "license": "MIT",
      "homepage": "https://github.com/paulmeier/kasas-plugins/tree/main/plugins/coffee-budget",
      "runtime": "lua",                       // "lua" | "js"
      "entrypoint": "main.lua",
      "hooks": ["OnTransactionCreate", "OnTransactionUpdate"],
      "capabilities": ["transactions:read", "labels:write"],
      "capability_tier": "write",             // "read-only" | "write"
      "ui": { "title": "Coffee Budget", "icon": "coin" },  // present iff the plugin has a dashboard page
      "path": "plugins/coffee-budget",        // repo-relative directory
      "files": [
        { "path": "README.md",   "sha256": "…", "size": 776 },
        { "path": "main.lua",    "sha256": "…", "size": 823 },
        { "path": "plugin.toml", "sha256": "…", "size": 552 }
      ],
      "content_hash": "sha256:…",             // aggregate over all files
      "size_bytes": 2151
    }
  ]
}
```

Plugins are sorted by `name`, and each plugin's `files` are sorted by `path`, so the
document is byte-stable across machines (only `generated_at` varies, which the
`--check` comparison ignores).

## Field reference

| Field | Meaning |
| ----- | ------- |
| `schema_version` | Format version. A client should refuse a version it doesn't understand. |
| `generated_at` | When the index was built (UTC). Informational; not part of the integrity contract. |
| `repository` | The registry repository, the base for resolving `path`. |
| `plugins[].path` | Repo-relative plugin directory. Combine with `repository` + a ref to fetch files (e.g. `raw.githubusercontent.com/.../main/<path>/<file>`). |
| `plugins[].files[]` | Every file the installer must download: `path` (plugin-relative, forward slashes), `sha256` (hex), `size`. |
| `plugins[].content_hash` | `sha256:` + the SHA-256 of `"<path>\0<filehash>\n"` lines in `path` order. Changes if any file's name or content changes — use it to detect updates. |
| `plugins[].capability_tier` | `read-only` if the plugin only reads; `write` if it can modify user data. The dashboard should warn more prominently on `write`. |
| `plugins[].ui` | Present only when the plugin contributes a dashboard page: its sidebar `title` and curated `icon` name. The marketplace can badge such plugins ("adds a dashboard page") before install. |

## How the dashboard should install a plugin

1. Fetch and parse the index; check `schema_version`.
2. Present the catalog. For each plugin show `description`, `author`, `runtime`,
   `hooks`, `capabilities`, and surface `capability_tier` (warn on `write`).
3. On install, download each file in `files[]` from
   `<repository>/raw/<ref>/<path>/<file.path>` and verify its `sha256` **before**
   writing it.
4. Optionally recompute and compare `content_hash` as a whole-plugin integrity
   check.
5. Write the verified files into `plugins.dir/<name>/`. The host discovers the
   plugin **disabled**; enabling it (which runs its code) stays an admin action, per
   the host's opt-in model.

The hashes are the chain of custody from "reviewed in this repo" to "running on the
user's machine": the dashboard never has to trust the transport, only the index it
fetched.
